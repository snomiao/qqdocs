# qqdocs UX improvement notes — 2026-04-19
> Cross-pollinated from onenote-cli comparison

## Learn from onenote-cli

### 1. Path-style navigation in `ls`
onenote: `onenote ls "MyNotebook/Section"` — depth inferred from path segments.
qqdocs: `qqdocs ls <folder-id>` — user must know the raw ID.

**Idea:** support `qqdocs ls "项目文档/2026"` by resolving each path segment with
`manage.folder_list` recursively. Same uniqueness rule as name resolution in `resolveFileId`.

### 2. Richer `<ref>` epilog
onenote explains path/ID/URL with examples AND a uniqueness warning.
qqdocs epilog is one terse line.

**Idea:** expand to:
```
<ref> accepts any of:
  - name     e.g. '账本'  (unique match required — use ID/URL if ambiguous)
  - ID       e.g. 'DZFJhVkJHSUdJcENR'
  - URL      e.g. 'https://docs.qq.com/sheet/DZFJhVkJHSUdJcENR'
```

### 3. `--sha` naming for delete confirm
onenote uses `--sha` (short, memorable). qqdocs uses `--confirm`.
`--confirm` is more self-explanatory for two-step flow; keep it.
But alias `--sha` could be added for muscle memory from onenote users.

### 4. `sync` / local cache for search
onenote caches doc index locally → instant search.
qqdocs hits network every time.

**Idea:** `qqdocs sync` writes `~/.qqdocs/cache.json` (title+id+url+mtime).
`qqdocs search` checks cache first, falls back to remote.
Cache invalidated by `--refresh` or TTL.

### 5. `mv` on onenote infers resource type from path depth
qqdocs `mv` is rename-only. Could extend to "move into folder" if dst looks like a folder ref.

## What qqdocs already does better

- Real-time remote search (no sync required)
- `import` local files → cloud
- `perm get/set` permission management
- `space` knowledge-base tree management
- `canvas` block-level editing
- OSC 8 hyperlinks + `.ext` suffix in TTY output
- `--dates` parallel fetch
- `create` with content/format/perm in one shot
